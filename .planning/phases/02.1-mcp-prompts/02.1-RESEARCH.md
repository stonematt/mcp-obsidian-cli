# Phase 2.1: MCP Prompts - Research

**Researched:** 2026-04-03
**Domain:** MCP prompt registration, Node.js file loading, MIT content adaptation
**Confidence:** HIGH

## Summary

Phase 2.1 adds four static MCP prompts to the existing single-file server and enriches tool descriptions. The technical implementation is straightforward: the `@modelcontextprotocol/sdk` `McpServer` class already supports `server.prompt()` with the exact same ergonomics as `server.tool()`. Calling `server.prompt()` auto-registers the `prompts` capability — no manual capabilities update needed. Content lives in external `.md` files loaded with `readFileSync` at startup (already imported), preserving the zero-build-step constraint.

The source material (Kepano's obsidian-skills repo, MIT licensed) has been fetched and reviewed. All four SKILL.md files exist at `skills/{obsidian-cli,obsidian-markdown,obsidian-bases,json-canvas}/SKILL.md`. Content is suitable for adaptation: it is already compact, reference-oriented, and written for AI agents. The main adaptation work is adding tool-integration tips sections and removing plugin-development content from `obsidian-cli` that doesn't apply to MCP consumers.

The only subtle implementation detail: the `GetPromptResult` the callback must return has shape `{ description?, messages: [{role, content}] }` where `content` is a standard MCP content block (`{type: "text", text: "..."}`) and `role` is `"user"` or `"assistant"`. For static reference prompts, `role: "user"` is the canonical choice.

**Primary recommendation:** Use `server.prompt(name, description, cb)` (deprecated but stable) or `server.registerPrompt(name, config, cb)` (preferred). Load `.md` content at module top-level with `readFileSync`. Add `prompts/` to `package.json` `files` array. Do not add prompts capability to the `McpServer` constructor — it is auto-registered on first `server.prompt()` call.

### Pattern 1: Prompt File Loading at Module Top-Level

**What:** Load all prompt `.md` files once at startup using `readFileSync`, before any registrations.
**When to use:** Always — synchronous reads at module load time are acceptable for small config/content files.

```javascript
// Source: node:fs (already imported in server.js)
import { readFileSync } from "node:fs";
import { fileURLToPath } from "node:url";
import { dirname, join } from "node:path";

const __dirname = dirname(fileURLToPath(import.meta.url));
const PROMPTS_DIR = join(__dirname, "prompts");

const promptContent = {
  "obsidian-cli": readFileSync(join(PROMPTS_DIR, "obsidian-cli.md"), "utf8"),
  "obsidian-markdown": readFileSync(join(PROMPTS_DIR, "obsidian-markdown.md"), "utf8"),
  "obsidian-bases": readFileSync(join(PROMPTS_DIR, "obsidian-bases.md"), "utf8"),
  "obsidian-canvas": readFileSync(join(PROMPTS_DIR, "obsidian-canvas.md"), "utf8"),
};
```

**Important:** `server.js` already imports `readFileSync` and `join` but NOT `dirname` or `fileURLToPath`. Both `__dirname` emulation imports must be added. `import.meta.url` is available because the project uses ES modules (`"type": "module"`).

### Pattern 2: Prompt Registration

**What:** Call `server.registerPrompt()` (preferred) or `server.prompt()` (deprecated but functional) after tool registrations.
**When to use:** For each of the 4 prompts.

```javascript
// Source: node_modules/@modelcontextprotocol/sdk/dist/esm/server/mcp.js
server.registerPrompt(
  "obsidian-cli",
  {
    title: "Obsidian CLI Reference",
    description: "CLI usage patterns, parameter syntax, and command examples for the Obsidian CLI. Adapted from kepano/obsidian-skills (MIT).",
  },
  () => ({
    messages: [
      {
        role: "user",
        content: { type: "text", text: promptContent["obsidian-cli"] },
      },
    ],
  })
);
```

**Key facts from SDK source:**
- Calling `server.registerPrompt()` or `server.prompt()` automatically calls `this.server.registerCapabilities({ prompts: { listChanged: true } })` — do NOT add `prompts: {}` to the `McpServer` constructor's `capabilities` object.
- The callback for a no-args prompt receives only `extra` (the request handler extra), not an args object.
- `GetPromptResult` shape: `{ description?: string, messages: PromptMessage[] }`
- `PromptMessage` shape: `{ role: "user" | "assistant", content: ContentBlock }`
- `ContentBlock` for text: `{ type: "text", text: string }`

### Pattern 4: package.json files Array Update

**What:** Add `prompts/` to the npm `files` array so `.md` content ships with the package.

```json
{
  "files": [
    "server.js",
    "prompts/"
  ]
}
```

**Note:** The current `package.json` has no `files` array — npm defaults to shipping everything not in `.npmignore`. Adding `files` makes it explicit and prevents accidentally shipping test files, planning dirs, etc.

### Anti-Patterns to Avoid

- **Adding `prompts: {}` to the McpServer constructor:** The SDK auto-registers this capability when `server.prompt()` or `server.registerPrompt()` is first called. Adding it manually is harmless but redundant. The CONTEXT.md mentions it was needed — confirmed by SDK source that it is not.
- **Loading prompt files lazily inside the callback:** Load at startup. If a file is missing, fail fast at boot time rather than silently returning empty content when the prompt is called.
- **Async file loading:** `readFileSync` at module top-level is correct for small content files. No need for `fs.promises.readFile`.
- **Using `__dirname` directly:** ES modules don't have `__dirname`. Use `dirname(fileURLToPath(import.meta.url))`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Prompt registration | Custom handlers for `prompts/list` and `prompts/get` | `server.registerPrompt()` | SDK handles capability registration, validation, error codes, argument parsing |
| Content at runtime | Dynamic file reading in callback | `readFileSync` at module load | Simpler, fails fast if files missing, no async complexity |

**Key insight:** The MCP SDK's `McpServer.registerPrompt()` handles everything — capability advertisement, list response, get-with-args dispatch, error codes for missing prompts. There is nothing to implement beyond calling it and providing the callback.

## Common Pitfalls

### Pitfall 1: `__dirname` not defined in ES modules

**What goes wrong:** `readFileSync(__dirname + "/prompts/...")` throws `ReferenceError: __dirname is not defined`.
**Why it happens:** ES module context (required by MCP SDK). `__dirname` is CommonJS only.
**How to avoid:** Use `dirname(fileURLToPath(import.meta.url))` to reconstruct equivalent.
**Warning signs:** Any use of `__dirname` directly in a `.js` file with `"type": "module"` in package.json.

### Pitfall 2: Missing prompt files crash at startup

**What goes wrong:** If a `.md` file is missing from the `prompts/` directory, `readFileSync` throws at module load and the server crashes before registering any tools.
**Why it happens:** Synchronous load at top level propagates errors immediately.
**How to avoid:** Create all four `.md` files before testing. Or wrap reads in try/catch with a fallback string — but failing fast is preferable for a packaged server.
**Warning signs:** `ENOENT` error referencing a prompts path at server startup.

### Pitfall 3: Prompts directory missing from npm package

**What goes wrong:** `npx mcp-obsidian-cli` works locally but crashes for end users with `ENOENT` on the prompt `.md` files.
**Why it happens:** npm's default inclusion rules may exclude the `prompts/` directory, or the `files` array may not include it.
**How to avoid:** Add `"prompts/"` to the `files` array in `package.json`. Verify with `npm pack --dry-run`.
**Warning signs:** File paths resolve correctly during local dev but fail in a temp directory installation.

### Pitfall 4: Capabilities object in McpServer constructor is not needed

**What goes wrong:** Planner adds `capabilities: { tools: {}, prompts: {} }` to the McpServer constructor, then the SDK double-registers the prompts capability when `server.registerPrompt()` is called.
**Why it happens:** CONTEXT.md mentioned adding `prompts: {}` to capabilities, but SDK source confirms auto-registration.
**How to avoid:** Leave the McpServer constructor as-is (`capabilities: { tools: {} }`). The SDK handles prompts capability registration automatically.
**Warning signs:** No actual error, but unnecessary code.
