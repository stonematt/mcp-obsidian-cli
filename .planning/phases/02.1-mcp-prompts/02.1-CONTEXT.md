<decisions>
## Implementation Decisions

### Prompt Content Source
- **D-01:** Adapt Kepano's obsidian-skills content for MCP context ‚Äî use as foundation but rewrite for an AI assistant audience, focusing on what an LLM needs to correctly use the CLI and Obsidian markdown
- **D-02:** Each prompt includes tool integration tips section tying knowledge back to this server's tools (e.g., "Use obsidian('search:context query=...') to find notes matching...")
- **D-03:** MIT attribution in prompt metadata/description only (visible in `prompts/list`), not in the actual prompt response content

### Prompt Structure
- **D-04:** Static prompts with no arguments ‚Äî `get_prompt('obsidian-cli')` returns the full reference document
- **D-05:** One prompt per topic ‚Äî 4 prompts matching requirements: `obsidian-cli`, `obsidian-markdown`, `obsidian-bases`, `obsidian-canvas`

### Tool Description Enrichment
- **D-06:** Expand inline `server.tool()` description strings with parameter details and 1-2 usage examples per tool
- **D-07:** CLI help text manually curated from `obsidian help <command>` output ‚Äî static strings, no runtime overhead

### Content Organization
- **D-08:** Prompt content in external `.md` files in a `prompts/` directory, loaded with `readFileSync` at startup
- **D-09:** Ship `.md` files in npm package (add `prompts/` to package files) ‚Äî zero build step preserved

### Claude's Discretion
- Exact prompt content structure and section ordering within each .md file
- How much of Kepano's content to include vs trim for MCP relevance
- Specific CLI help text excerpts to include in tool descriptions
